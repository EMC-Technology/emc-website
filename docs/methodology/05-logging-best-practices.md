## 一、引言

### 1.1 文档目的

本文档阐述日志记录策略的设计方法论，提供结构化日志设计的原则和最佳实践，帮助开发团队建立有效的日志记录体系。

### 1.2 日志的核心价值

日志在系统中的核心作用体现在四个维度：

**调试与诊断 (Debugging)**
- 开发阶段追踪程序执行流程
- 生产环境问题定位和根因分析
- 历史操作追溯和复现

**监控与告警 (Monitoring & Alerting)**
- 实时系统状态监控
- 异常情况自动告警
- 性能指标采集

**审计与合规 (Audit & Compliance)**
- 安全事件追踪
- 操作行为审计
- 合规要求满足

**分析与洞察 (Analytics & Insights)**
- 用户行为分析
- 系统使用趋势
- 错误模式识别

## 二、日志级别设计

### 2.1 级别定义框架

日志级别应与错误的严重程度形成映射：

**ERROR (错误级别)**
- 定义：程序运行中发生的错误事件
- 影响：某个功能无法正常工作，但系统整体仍可运行
- 内容：完整的错误信息，包括异常堆栈、上下文
- 示例：数据库连接失败、API调用超时

**WARN (警告级别)**
- 定义：潜在问题或不良状态的警告
- 影响：操作可能受影响，但有替代方案
- 内容：问题摘要，不含堆栈跟踪
- 示例：资源使用率偏高、配置项废弃

**INFO (信息级别)**
- 定义：重要的正常操作事件
- 影响：无负面作用，仅作为系统运行状态的参考
- 内容：关键业务事件和状态变化
- 示例：服务启动成功、任务完成确认

**DEBUG (调试级别)**
- 定义：开发调试用的详细信息
- 影响：仅在开发/调试环境启用
- 内容：详细执行轨迹、变量值
- 示例：函数入口/出口、循环次数

**TRACE (追踪级别)**
- 定义：最详细的执行轨迹
- 影响：仅在需要深入分析时临时启用
- 内容：每一行的执行轨迹
- 示例：SQL语句、详细参数

### 2.2 级别映射关系

| 错误严重程度 | 日志级别 | 记录策略 |
|-------------|----------|----------|
| CRITICAL (致命) | ERROR | 立即记录 + 触发告警 |
| ERROR (错误) | ERROR | 始终记录完整错误信息 |
| WARNING (警告) | WARN | 记录摘要，不含堆栈 |
| INFO (信息) | INFO | 按采样率记录 |

## 三、结构化日志设计

### 3.1 结构化日志概念

结构化日志将日志信息组织为键值对形式，便于检索和分析。

**传统日志格式**：
```
2026-04-05 10:15:23 ERROR [ai_model] AI模型调用失败: API速率限制 (HTTP 429)
```
问题：难以解析和检索，难以进行统计分析，难以与其他系统集成。

**结构化日志格式 (JSON)**：
```json
{
  "timestamp": "2026-04-05T10:15:23.456Z",
  "level": "ERROR",
  "target": "ai_model::lm_manager",
  "error_code": "ERR-AIM-LM-002_ERR_S",
  "error_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "message": "AI模型 GPT-5 调用失败: API速率限制 (HTTP 429)",
  "structured_data": {
    "model": "GPT-5",
    "http_status": 429,
    "retry_after_seconds": 60
  }
}
```
优势：易于解析和检索，支持复杂查询和统计，便于与日志分析系统集成。

### 3.2 日志字段设计

**必填字段 (Must Have)**：

1. **timestamp**: 日志产生时间
   - ISO8601 格式
   - UTC 时区
   - 毫秒精度

2. **level**: 日志级别
   - ERROR / WARN / INFO / DEBUG / TRACE

3. **target**: 日志目标模块
   - 模块路径，如 "ai_model::lm_manager"

4. **message**: 日志消息
   - 人类可读的消息文本
   - 长度限制: 4096 字符

**条件必填字段 (Should Have)**：

1. **error_code**: 标准错误码
   - 当 level ∈ {ERROR, WARN} 时必填

2. **error_id**: 错误实例ID
   - 用于关联同一次错误的多个日志条目

3. **session_id / request_id**: 关联ID
   - 用于追踪特定会话或请求的所有日志

**可选字段 (Nice to Have)**：

1. structured_data: 结构化附加数据
2. span_id: 分布式追踪ID
3. file / line: 源码位置
4. thread_name: 线程名称
5. hostname: 主机名

## 四、日志存储策略

### 4.1 分类存储架构

不同类型的日志应有不同的存储策略：

**errors.log (一般错误日志)**
- 用途：记录 ERROR 和 WARN 级别错误
- 保留期: 30天
- 大小上限: 2GB
- 特点：标准轮转策略

**critical.log (致命错误日志)**
- 用途：记录 CRITICAL 级别错误
- 保留期: 90天
- 大小上限: 500MB
- 特点：独立存储，高优先级保护

**audit.log (审计日志)**
- 用途：记录安全相关的操作和事件
- 保留期: 180天
- 大小上限: 10GB
- 特点：追加写入，HMAC完整性校验

**performance.log (性能日志)**
- 用途：记录性能指标和慢操作
- 保留期: 7天
- 大小上限: 5GB
- 特点：缓冲写入，减少I/O

**debug.log (调试日志)**
- 用途：记录 DEBUG/TRACE 级别信息
- 保留期: 3天
- 大小上限: 500MB
- 特点：默认关闭，需要时启用

### 4.2 轮转策略设计

**轮转策略类型对比**：

| 策略类型 | 轮转触发条件 | 适用场景 |
|----------|-------------|----------|
| 按大小轮转 | 单文件达到大小上限 | 高吞吐量、日志量波动大的系统 |
| 按时间轮转 | 到达预定时间点 | 需要按日/周/月归档的场景 |
| 混合轮转 | 同时满足大小或时间 | 兼具两者优势（推荐） |

**推荐: 混合轮转策略**

- **主策略: 按日轮转 (时间驱动)**
  - 每日 00:00 UTC 创建新文件
  - 文件名格式: errors-2026-04-05.log

- **子策略: 大小兜底 (容量保护)**
  - 单文件超过 100MB 时创建额外文件
  - 文件名格式: errors-2026-04-05.log.1

- **清理策略: 自动清理**
  - 保留最近 30 天
  - 或总大小超过 2GB 时清理最旧文件

## 五、日志记录最佳实践

### 5.1 记录原则

**原则 1: 记录必要信息**
- 记录足够诊断问题的信息
- 不要记录无关的冗余信息
- 平衡信息完整性和性能开销

**原则 2: 避免敏感信息泄露**
- 脱敏处理: API Key → "****1234"
- 脱敏处理: 密码 → "******"
- 路径处理: /home/user/... → "..."

**原则 3: 使用适当的日志级别**
- ERROR: 真正的错误
- WARN: 潜在问题
- INFO: 重要业务事件
- DEBUG: 开发调试信息

**原则 4: 结构化记录**
- 使用键值对组织信息
- 便于检索和分析
- 使用统一字段命名

**原则 5: 包含上下文信息**
- 包含 error_id 用于关联
- 包含 session_id / request_id
- 包含足够的调试信息

### 5.2 常见问题

**问题 1: 敏感信息泄露**
- BAD: log.error("API调用失败: key={}", api_key);
- GOOD: log.error("API调用失败: key=****{}",
                   api_key.last_chars(4));

**问题 2: 日志过多**
- 问题: 循环中的高频日志导致性能问题
- 解决: 使用采样或降级策略

**问题 3: 信息不足**
- BAD: log.error("操作失败");
- GOOD: log.error("文件保存失败: path={}, error={}",
                   path, io_error);

**问题 4: 异步日志故障丢失**
- 问题: 异步日志写入失败可能丢失日志
- 解决: 使用有界队列，失败时降级到同步写入

## 六、安全日志设计

### 6.1 安全日志要点

安全日志必须记录的内容：

1. **认证事件**
   - 登录成功/失败
   - 令牌刷新/过期
   - 权限变更

2. **授权事件**
   - 访问被拒绝
   - 权限提升操作

3. **安全策略事件**
   - 密码修改
   - MFA变更
   - 安全策略违规

4. **可疑活动**
   - 多次登录失败
   - 异常访问模式
   - 敏感数据访问

### 6.2 审计日志保护

**完整性保护**：
- 使用 HMAC 对每条日志进行签名
- 检测日志篡改或删除
- 支持日志完整性验证

**访问控制**：
- 仅管理员可删除日志
- 日志访问有审计追踪
- 分离存储防止被篡改

## 七、总结

日志记录策略是系统可观测性的基础。一个好的日志系统应该：

1. **结构化**：便于解析、检索和分析
2. **分级清晰**：正确使用日志级别
3. **信息完整**：包含足够的上下文信息
4. **安全合规**：脱敏处理，保护敏感信息
5. **可持续**：有合理的轮转和清理策略
6. **可追踪**：支持分布式场景下的关联分析

本文档提供的日志设计原则是通用指导，具体实现应根据系统规模、存储成本和合规要求进行调整。
