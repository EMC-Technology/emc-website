## 一、引言

### 1.1 文档目的

本文档阐述错误码格式设计的方法论和规范，提供通用的错误码格式设计原则和最佳实践，适用于各类软件系统的错误码体系设计。

### 1.2 错误码的核心价值

错误码在系统中的核心作用包括：

1. **全局唯一标识**：每个错误都有一个全局唯一的代码，便于精确引用和追踪

2. **语义载体**：错误码的结构蕴含了错误的关键属性信息，可以快速判断错误类型、来源、严重程度等

3. **可检索性**：用户和开发者可以通过错误码快速检索相关文档和解决方案

4. **可分析性**：错误码便于进行统计分析和趋势预测，支持按维度聚合和问题定位

## 二、错误码格式设计原则

错误码格式设计应遵循以下六项核心原则：

**可读性优先**
- 错误码应该易于人类阅读和理解
- 结构化设计使错误码的含义一目了然
- 避免使用无意义的数字或随机字符串

**可解析性**
- 错误码结构应该规则化，便于程序解析
- 正则表达式可以完整验证格式
- 支持按字段提取和过滤

**可扩展性**
- 格式设计应预留扩展空间
- 新增错误类型时不需要修改已有格式
- 保持向后兼容性

**全局唯一性**
- 每个错误码在整个系统内必须全局唯一
- 避免不同模块使用相同代码表示不同错误
- 建立统一的错误码注册中心

**语义一致性**
- 错误码的结构应与分类体系对应
- 错误码的各部分应与分类维度一致
- 便于从错误码推断错误属性

**稳定性**
- 已发布的错误码语义不得变更
- 废弃的错误码应保留标记而非删除
- 确保长期兼容性和文档有效性

## 三、错误码格式结构

### 3.1 通用格式模型

错误码格式应包含以下结构化组成部分：

```
标准格式: ERR-[来源]-[模块]-[序号]_[严重程度][影响范围]

示例: ERR-AIM-LM-002_ERR_S

各部分说明：
• ERR: 固定前缀，表明这是一个错误码
• 来源: 标识错误产生的来源分类（2-4字符大写字母）
• 模块: 标识错误归属的模块（2-4字符大写字母）
• 序号: 模块内的错误序号（3位数字）
• 严重程度: 错误严重程度等级
• 影响范围: 错误的影响范围
```

### 3.2 格式字段定义

| 字段 | 长度 | 字符集 | 说明 |
|------|------|--------|------|
| 前缀 | 固定 | `ERR` | 标识错误码类型 |
| 来源分隔符 | 固定 | `-` | 分隔前缀与来源 |
| 来源标识 | 2-4字符 | 大写字母 | 错误来源分类 |
| 模块标识 | 2-4字符 | 大写字母 | 所属模块 |
| 序号 | 3位数字 | `0-9` | 模块内序号 |
| 严重程度分隔符 | 固定 | `_` | 分隔序号与严重程度 |
| 严重程度 | 3-4字符 | `CRI/ERR/WRN/INF` | 严重程度等级 |
| 影响范围 | 1字符 | `G/S/O/M` | 影响范围代码 |

### 3.3 格式验证规则

错误码格式应满足以下验证规则：

**BNF 语法定义**：
```
error_code ::= 'ERR' '-' source '-' module '-' sequence '_' severity scope
source     ::= ALPHA{2,4}
module     ::= ALPHA{2,4}
sequence   ::= DIGIT{3}
severity   ::= 'CRI' | 'ERR' | 'WRN' | 'INF'
scope      ::= 'G' | 'S' | 'O' | 'M'
```

**正则表达式**：
```
^ERR-[A-Z]{2,4}-[A-Z]{2,4}-\d{3}_(CRI|ERR|WRN|INF)_[GSOM]$
```

**验证检查项**：
1. 前缀必须为 "ERR"
2. 来源标识必须是在册的来源分类
3. 模块标识必须是在册的模块标识
4. 序号必须在有效范围内
5. 严重程度必须是有效等级
6. 影响范围必须是有效代码
7. 各部分长度必须符合规范

## 四、错误码分配规范

### 4.1 序号分段方案

序号应按照功能进行分段，便于管理和检索：

| 序号段 | 用途说明 |
|--------|----------|
| 001-099 | 初始化与配置相关错误：系统启动初始化失败、配置文件加载错误、依赖服务连接失败 |
| 100-199 | 输入验证与参数校验错误：用户输入格式错误、参数范围超限、必填参数缺失 |
| 200-299 | 核心功能逻辑错误：业务规则违反、状态转换非法、业务数据不一致 |
| 300-399 | 状态管理与生命周期错误：状态机非法转换、资源状态异常、生命周期事件错误 |
| 400-499 | 通信与协议交互错误：网络连接失败、协议解析错误、数据传输错误 |
| 500-599 | 数据持久化与 I/O 错误：文件读写错误、数据库操作失败、缓存访问错误 |
| 600-699 | 安全与权限错误：认证失败、授权拒绝、安全策略违规 |
| 700-799 | 性能与资源错误：资源耗尽、超时错误、负载过高 |
| 800-899 | 扩展与插件错误：插件加载失败、扩展版本不兼容、扩展依赖缺失 |
| 900-999 | 保留/预留：未来扩展使用、特定场景预留 |

### 4.2 模块标识分配原则

模块标识应遵循以下原则：

1. **唯一性**：每个模块标识必须唯一，不能重复
2. **稳定性**：模块标识确定后不可变更
3. **可读性**：标识应有意义，便于理解和记忆
4. **简洁性**：2-4个字符，简洁明了
5. **层次性**：按架构层次分组

**层次分组示例**：
- 应用层模块：UI（用户界面）、GW（网关服务）
- 服务层模块：LM（AI模型管理）、SS（会话管理）、TL（工具系统）、MC（MCP集成）、CF（配置管理）
- 集成层模块：LS（LSP客户端）
- 扩展层模块：PL（插件系统）
- 安全层模块：SEC（安全模块）

## 五、错误码生命周期管理

### 5.1 生命周期状态机

错误码生命周期包含四个状态：

**DRAFT**：新错误码的初始状态，需审核通过后方可使用

**ACTIVE**：正式使用的错误码，语义不得变更

**DEPRECATED**：已废弃但未删除，至少保留2个主版本后才可移除

**REMOVED**：已删除，不再使用

### 5.2 生命周期管理规则

**DRAFT 状态规则**：
- 新错误码在正式发布前必须经过审核
- 必须记录完整的元数据（含义、解决方案、影响范围）
- DRAFT 状态的错误码不能在实际运行中使用

**ACTIVE 状态规则**：
- 已发布的错误码语义不得变更
- 仅可添加新的错误码，不能修改已有错误码
- 必须保留完整的文档和解决方案

**DEPRECATED 状态规则**：
- 必须保留原语义和文档
- 应在文档中标注废弃原因和替代方案
- 至少保留2个主版本后才可移除
- 新代码不应使用已废弃的错误码

**REMOVED 状态规则**：
- 错误码完全从系统中删除
- 相关的文档应标注为历史版本
- 原码点不得再被使用

## 六、错误码设计最佳实践

### 6.1 命名规范

**推荐做法**：
- 使用有意义的来源和模块标识
  - GOOD: ERR-USR-001_ERR_S（用户输入错误）
  - GOOD: ERR-NET-001_ERR_O（网络错误）
- 序号应按功能分段
  - GOOD: ERR-USR-101_***（输入验证错误）
  - GOOD: ERR-USR-201_***（业务验证错误）
- 严重程度应与实际影响匹配
  - GOOD: ERR-FS-003_CRI_G（磁盘严重不足）

**避免做法**：
- 避免使用无意义的序号
  - BAD: ERR-XXX-001_ERR_S
  - BAD: ERR-XXX-999_ERR_S
- 避免随意分配序号
  - BAD: ERR-USR-500_ERR_S（用户错误用500段）
- 避免一个错误码表示多种含义
  - BAD: 同一错误码在不同场景有不同语义

### 6.2 文档要求

每个错误码都应配有完整的文档，包含以下字段：

1. 错误码: [完整错误码]
2. 名称: [简短的描述性名称]
3. 严重程度: [CRI/ERR/WRN/INF]
4. 影响范围: [G/S/O/M]
5. 可恢复性: [AUTO/SEMI/MANUAL/NON]
6. 含义: [详细的错误含义描述]
7. 可能原因: [列出常见原因]
8. 解决方案: [提供解决步骤]
9. 预防措施: [如何避免此错误]
10. 相关错误码: [相关的其他错误码]

## 七、总结

错误码格式规范是错误处理系统的基础组件。一个好的错误码格式应该：

1. **结构清晰**：格式各部分含义明确，易于理解
2. **可解析**：便于程序自动处理和分析
3. **可扩展**：预留扩展空间，适应系统演进
4. **易管理**：有完善的生命周期管理机制
5. **文档完善**：每个错误码都有完整的文档支持

本文档提供的格式规范是通用指导原则，具体实现时应根据项目规模、团队习惯和业务需求进行调整。
