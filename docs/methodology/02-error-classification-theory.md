## 一、引言

### 1.1 文档目的

本文档阐述软件错误分类的理论体系，提供错误分类的维度设计原则和分类方法论，帮助架构师和开发者建立科学的错误分类体系。

### 1.2 为什么需要错误分类

错误分类是建立有效错误处理系统的基础。缺乏分类的错误处理会导致：所有错误混合在一起难以分析、无法制定针对性恢复策略、日志和监控无法有效组织、用户面对大量无意义的错误码。

科学的分类体系能够带来：错误有清晰的语义易于理解、可以针对不同类型制定不同策略、便于统计分析和趋势预测、提升用户体验和运维效率。

## 二、错误分类维度理论

### 2.1 多维分类概念

单一维度的分类往往不足以描述错误的完整特征，需要采用多维度分类体系。四维分类空间通过来源、严重程度、影响范围、可恢复性四个维度组合，形成错误的完整分类指纹。

理论组合数为 N₁ × N₂ × N₃ × N₄，实际有效组合数根据约束条件缩减。

### 2.2 维度设计原则

每个分类维度都应满足以下设计原则：

**互斥性 (Mutually Exclusive)**
- 每个维度内的取值应该是互斥的，一个错误实例的每个维度只能取一个值
- 避免歧义，简化处理逻辑

**完备性 (Collectively Exhaustive)**
- 维度取值应该覆盖所有可能的错误场景
- 对于无法预见的场景，应有"未知"或"其他"作为兜底

**独立性 (Independence)**
- 各维度之间应该相互独立，一个维度的取值不应影响另一维度的取值
- 便于独立演化和扩展

**可判定性 (Decidability)**
- 每个维度的取值应该是可判定的，有明确的标准和流程来确定
- 避免主观判断导致的分类不一致

**可操作性 (Actionability)**
- 每个维度的取值应该能指导具体的处理行为
- 分类的目的最终还是为了更好地处理错误

## 三、核心分类维度

### 3.1 维度一：错误来源

错误来源维度标识错误产生的位置或原因。按来源性质可分为三类：外部输入类（来自系统外部的输入，如用户输入、外部API响应）、外部依赖类（外部服务和资源，如网络、文件系统、第三方服务）、内部系统类（系统内部的逻辑和状态）。

设计要点：来源应该与系统架构层次对应；一个错误应该只能有一个主要来源；对于混合来源的错误，取最根本的原因作为来源。

### 3.2 维度二：严重程度

严重程度维度标识错误的紧急程度和影响。四级严重程度模型如下：

**级别 1: 致命 (Critical)**
- 定义：系统无法继续运行，必须立即停止
- 影响：数据安全或系统完整性受到威胁
- 处理：安全停机，保存现场，通知运维
- 示例：核心数据损坏、安全密钥泄露

**级别 2: 错误 (Error)**
- 定义：当前操作失败，但系统整体仍可运行
- 影响：功能受损，但不影响其他操作
- 处理：根据错误类型尝试恢复或降级
- 示例：文件操作失败、网络请求超时

**级别 3: 警告 (Warning)**
- 定义：操作成功但存在潜在问题
- 影响：可能影响后续操作或性能
- 处理：记录日志，可能需要关注
- 示例：资源使用率偏高、配置项废弃

**级别 4: 信息 (Info)**
- 定义：正常操作的重要事件通知
- 影响：无负面作用，仅供参考
- 处理：记录日志，供监控和分析
- 示例：服务启动成功、操作完成确认

### 3.3 维度三：影响范围

影响范围维度标识错误影响的系统范围。四级影响范围模型如下：

**范围 1: 全局性 (Global)**
- 定义：影响整个系统或所有用户
- 处理优先级：最高 (P0)
- 示例：核心服务宕机、数据库连接失败

**范围 2: 会话级 (Session)**
- 定义：仅影响当前单个用户会话
- 处理优先级：高 (P1)
- 示例：用户认证失败、会话数据损坏

**范围 3: 操作级 (Operation)**
- 定义：仅影响当前单次原子操作
- 处理优先级：中 (P2)
- 示例：单次文件写入失败、单次API调用超时

**范围 4: 模块级 (Module)**
- 定义：仅影响特定功能模块
- 处理优先级：浮动 (P1-P2)
- 示例：LSP功能不可用、MCP工具调用失败

### 3.4 维度四：可恢复性

可恢复性维度标识错误的可恢复程度和方式。四类可恢复性模型如下：

**类型 1: 自动可恢复 (Auto Recoverable)**
- 定义：无需人工干预，可自动恢复
- 处理策略：立即重试 → 指数退避 → 优雅降级
- 用户感知：无或轻微延迟
- 适用：错误原因是瞬态的，如网络抖动、服务暂时不可用

**类型 2: 半自动恢复 (Semi-Auto)**
- 定义：需要用户确认选择后执行恢复
- 处理策略：展示选项面板，等待用户选择
- 用户感知：明确交互界面
- 适用：错误原因需要用户判断，如文件覆盖确认

**类型 3: 人工干预 (Manual Intervention)**
- 定义：需要开发者或运维人员介入
- 处理策略：诊断收集 → 报告生成 → 安全模式
- 用户感知：诊断报告 + 支持引导
- 适用：复杂问题需要专业知识，如配置错误、数据修复

**类型 4: 不可恢复 (Non-Recoverable)**
- 定义：数据损坏或不可逆不一致，无法恢复
- 处理策略：优雅停机 + 保存现场 + 引导联系支持
- 用户感知：系统退出提示
- 适用：严重数据损坏、安全违规等不可逆情况

## 四、分类一致性保证

### 4.1 正交性约束

维度之间应该保持正交性，但某些维度组合可能需要约束：

**规则 1: 严重程度"信息"不参与可恢复性分类**
- 原因：INFO 级别仅为通知性质，不涉及错误处理
- 效果：排除 1/4 的无效组合

**规则 2: 会话来源的范围固定为会话级**
- 原因：会话相关错误的范围本质上是会话级的
- 效果：简化会话相关错误的分类

**规则 3: 严重程度与可恢复性的倾向性关联**
- CRITICAL 倾向于 NON 或 MANUAL
- WARNING 倾向于 AUTO
- 这不是硬性约束，但提供了默认分类的指导

### 4.2 分类决策流程

分类判定流程包括以下步骤：

**步骤 1: 确定错误来源**
- 问题：错误的原始原因是什么？
- 输入：错误的现象、上下文、环境信息
- 输出：错误来源分类

**步骤 2: 确定严重程度**
- 问题：错误对系统运行的影响是什么？
- 输入：错误来源、系统状态、数据影响
- 输出：严重程度等级

**步骤 3: 确定影响范围**
- 问题：错误会影响哪些系统部分？
- 输入：错误的传播范围、共享资源
- 输出：影响范围分类

**步骤 4: 确定可恢复性**
- 问题：错误是否可恢复？如何恢复？
- 输入：错误原因、错误类型、系统状态
- 输出：可恢复性分类

**步骤 5: 验证分类结果**
- 检查：是否满足所有正交性约束？
- 检查：分类结果是否符合一致性要求？
- 如有问题，返回步骤1重新判定

## 五、分类体系评估指标

### 5.1 维度独立性评估

| 维度对 | 独立性评分 | 说明 |
|--------|-----------|------|
| 来源 vs 严重程度 | 9/10 | 弱相关，不同来源都可能出现各严重程度 |
| 来源 vs 影响范围 | 7/10 | 中等相关，某些来源倾向于特定范围 |
| 来源 vs 可恢复性 | 8/10 | 弱相关，取决于具体错误类型 |
| 严重程度 vs 影响范围 | 8/10 | 弱相关，但有倾向性关联 |
| 严重程度 vs 可恢复性 | 6/10 | 中等相关，有一定的关联倾向 |
| 影响范围 vs 可恢复性 | 7/10 | 弱相关，取决于错误类型 |

### 5.2 组合覆盖率评估

组合覆盖率计算公式：理论组合数 = 来源类别数 × 严重程度数 × 影响范围数 × 可恢复性数

假设来源类别数=15，严重程度数=4，影响范围数=4，可恢复性数=4，则理论组合=15×4×4×4=960种。有效组合数约380~420种（考虑正交性约束后），组合覆盖率约40%~44%。

质量评估标准：≥40%为良好，分类体系效率较高；30%-40%为可接受，存在优化空间；<30%为较差，分类维度可能存在问题。

## 六、总结

错误分类学理论是构建可靠错误处理系统的基础。一个好的错误分类体系应该：

1. **维度全面**：覆盖错误的所有重要特征
2. **取值互斥**：避免分类歧义
3. **体系完备**：覆盖所有可能的错误场景
4. **独立可扩展**：各维度独立，便于扩展
5. **指导行动**：分类结果能指导具体的处理决策

本文档提供的四维分类框架是一个通用的理论指导，具体实现时应根据项目实际情况进行调整，确定合适的维度数量和取值范围。
