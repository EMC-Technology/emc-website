## 一、引言

### 1.1 文档目的

本文档阐述用户界面错误提示设计的原则和方法论，提供错误提示的 UI/UX 设计指导，帮助设计师和开发者创建用户友好的错误提示体验。

### 1.2 错误提示的重要性

用户遇到错误时的典型心理状态会经历挫折感曲线。设计良好的错误提示可以缩短挫折感峰值、加速问题理解、帮助用户快速找到解决方案。

## 二、错误提示设计原则

### 2.1 六项核心原则

**原则 1: 清晰明确 (Clarity)**
- 使用通俗易懂的语言，避免技术术语
- 明确告知发生了什么问题
- 清楚说明问题的影响
- 提供具体的解决方向

**原则 2: 及时反馈 (Timeliness)**
- 尽早检测和报告错误
- 输入错误应即时反馈，不要等到提交后
- 异步操作应提供进度指示
- 错误恢复后应明确告知成功

**原则 3: 有用导向 (Helpfulness)**
- 提供可操作的解决方案
- 步骤应按优先级排序
- 复杂问题应提供更多帮助资源
- 避免指责用户

**原则 4: 适度打扰 (Respectfulness)**
- 根据严重程度选择合适的展示方式
- 轻微问题不要使用模态对话框
- 允许用户忽略非关键警告
- 避免重复相同的错误提示

**原则 5: 一致性 (Consistency)**
- 相似错误使用相似的展示方式
- 相同级别的错误使用相同的视觉样式
- 错误提示的位置和格式应保持一致
- 遵循产品的整体设计语言

**原则 6: 可访问性 (Accessibility)**
- 确保色盲用户能区分错误级别
- 提供文本替代方案
- 支持屏幕阅读器
- 保证键盘可操作

## 三、错误严重程度与展示方式映射

### 3.1 四级映射模型

错误提示的展示方式应根据严重程度进行分级：

**级别 1: 致命错误 (Critical)**
- 展示方式：模态对话框 + 全屏遮罩，红色醒目样式，唯一按钮："安全退出"
- 设计要点：明确告知"系统遇到严重问题"，显示错误参考码供技术支持查询，禁止展示技术细节，30秒无操作自动执行安全退出

**级别 2: 操作错误 (Error)**
- 展示方式：内联错误横幅，操作区域高亮，可折叠详情区
- 设计要点：用通俗语言解释原因，提供1-3个解决步骤，提供"查看详情"折叠区，包含"重试"/"跳过"/"取消"按钮

**级别 3: 警告提示 (Warning)**
- 展示方式：非侵入式提示条，黄色/橙色温和样式，自动消失计时器
- 设计要点：温和提醒不引起恐慌，说明潜在影响，提供"不再提示"选项，不阻断用户当前操作

**级别 4: 信息通知 (Info)**
- 展示方式：状态栏/通知区域，绿色/蓝色中性样式，静默记录模式
- 设计要点：仅在用户主动查看时可见，可选是否弹出通知，用于审计追踪和运维监控

### 3.2 交互阻断级别

| 严重程度 | 交互阻断级别 | 行为描述 |
|----------|-------------|----------|
| Critical | 完全阻塞 | 冻结所有用户输入，仅响应"安全退出"按钮 |
| Error | 区域阻塞 | 阻塞当前操作，允许切换到其他不受影响的区域 |
| Warning | 非阻塞 | 不拦截任何用户输入，超时后自动消失 |
| Info | 后台静默 | 无任何 UI 弹出，仅更新状态栏 |

## 四、错误消息设计

### 4.1 消息结构框架

错误消息应包含以下结构化元素：

1. **操作名称**：什么操作失败了
2. **问题描述**：发生了什么问题
3. **影响说明**：这个问题会怎样影响我
4. **解决建议**：我该如何解决这个问题
5. **参考信息**：在哪里可以获得更多帮助

### 4.2 模板变量系统

错误消息应支持动态变量，实现灵活的消息生成：

| 变量 | 含义 | 示例 | 用途 |
|------|------|------|------|
| {error_ref} | 标准错误参考码 | ERR-AIM-LM-002_ERR_S | 技术支持查询、错误追踪 |
| {operation} | 失败操作的友好名称 | "AI代码生成"、"文件保存" | 告知用户哪个操作失败了 |
| {reason} | 失败原因的通俗解释 | "网络连接不稳定" | 帮助用户理解问题原因 |
| {suggestion} | 解决建议 | "请检查网络后重试" | 指导用户解决问题 |
| {entity_name} | 涉及的具体实体名称 | "GPT-5"、"config.toml" | 提供具体上下文信息 |
| {doc_link} | 帮助文档链接 | https://docs.example.com/errors/ERR-001 | 提供更多信息获取途径 |

### 4.3 变量填充优先级

**优先级 1: 直接取值**
- 直接从 ErrorObject 字段取值
- 例如: error_ref ← ErrorObject.code

**优先级 2: 映射表查找**
- 根据错误码查表获取预定义文本
- 例如: reason ← ReasonMapping[error_code].reason

**优先级 3: 默认兜底**
- 使用通用默认文本
- 例如: reason ← "发生未知错误"

**优先级 4: 国际化转换**
- 根据用户语言设置进行翻译
- 例如: reason.zh-CN ← translate(reason, "zh-CN")

## 五、视觉设计规范

### 5.1 视觉层次

| 属性 | Critical | Error | Warning | Info |
|------|----------|-------|---------|------|
| 主色调 | #DC2626 (红) | #EF4444 (浅红) | #F59E0B (琥珀) | #3B82F6 |
| 背景色 | #FEF2F2 | #FEF2F2 | #FFFBEB | #EFF6FF |
| 图标 | ⛔ | ❌ / ✕ | ⚠️ / △ | ℹ️ / ● |
| 边框样式 | 3px双实线 | 2px单实线 | 2px虚线 | 1px细线 |
| 字体加粗 | 700 (Bold) | 600 (Semibold) | 500 (Medium) | 400 |
| 动效 | 脉冲闪烁 | 入场滑入弹跳 | 淡入淡出 | 无 |

### 5.2 无障碍设计

**色盲友好设计**：
- 除颜色外，使用图标/形状/文字同时标识
- Critical: ⛔ 红色圆形 + "严重" 文字
- Warning: ⚠️ 黄色三角形 + "警告" 文字
- Error: ❌ 红色叉号 + "错误" 文字
- Info: ℹ️ 蓝色圆形 + "信息" 文字

**屏幕阅读器支持**：
- 所有错误提示必须有替代文本
- 使用 ARIA role="alert" 标识动态错误
- 确保阅读顺序符合逻辑
- 焦点管理：自动聚焦到模态对话框

## 六、国际化设计

### 6.1 国际化原则

**规则 1: 所有用户可见消息必须支持国际化**
- 消息模板不以硬编码字符串存在
- 通过 message_key 索引翻译条目
- 模板变量在翻译中保持原位

**规则 2: 技术术语保留英文原文**
- Token, API Key, MCP, LSP 等保留英文
- 首次出现时可附中文注释

**规则 3: 默认语言跟随系统设置**
- 支持运行时热切换
- 缺失翻译时回退至英文

### 6.2 翻译组织结构

```
locales/
├── zh-CN.json    ← 简体中文（默认）
├── en-US.json    ← 美式英语（回退）
├── ja-JP.json    ← 日语
└── ko-KR.json    ← 韩语
```

JSON 结构应包含：locale 元数据、errors 对象中每个错误码对应的 operation/reason/suggestion 等字段。

## 七、最佳实践与反模式

### 7.1 推荐做法

1. 明确告知问题和解决方案
   - "文件保存失败: 磁盘空间不足。请删除不需要的文件后重试。"

2. 使用第二人称，语气友好
   - "请输入有效的邮箱地址" 而不是 "邮箱地址无效"

3. 提供具体可行的步骤
   - "1. 点击重试按钮\n2. 如果问题持续，请联系技术支持"

4. 避免暴露内部实现细节
   - 不要显示堆栈跟踪、文件路径、内部变量名

### 7.2 避免做法

1. 模糊笼统
   - "操作失败"、"发生错误"

2. 指责用户
   - "你输入了错误的值"、"由于你的操作导致..."

3. 技术术语堆砌
   - "Error code: 0x80070005, Access denied at path /var/data/..."

4. 难以执行的建议
   - "请联系系统管理员" （没有提供联系方式）

5. 过度重复
   - 相同错误反复弹出而不提供永久忽略选项

## 八、总结

用户界面错误提示设计是用户体验的重要组成部分。一个好的错误提示应该：

1. **清晰明确**：让用户一眼看懂问题所在
2. **及时反馈**：在错误发生时就及时告知
3. **有用导向**：提供可操作的解决方案
4. **适度打扰**：根据严重程度选择合适的展示强度
5. **一致统一**：保持视觉和交互的一致性
6. **无障碍**：确保所有用户都能获得有效信息

本文档提供的设计原则是通用指导，具体实现应根据产品特点、用户群体和品牌调性进行调整。
