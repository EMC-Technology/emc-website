## 一、引言

### 1.1 文档目的

本文档阐述软件错误恢复设计的方法论，提供错误恢复状态机、重试策略、优雅降级等设计模式，帮助开发者构建具有自我恢复能力的软件系统。

### 1.2 错误恢复的重要性

**用户体验角度**：
- 无恢复机制：用户操作失败 → 显示错误 → 用户手动重试 → 体验差
- 有恢复机制：操作失败 → 自动重试 → 成功 → 用户无感知 → 体验好

**系统可用性角度**：
- 自动恢复可以显著提高系统的可用性
- 特别是在处理瞬态错误时（如网络抖动、服务暂时不可用）

**运维效率角度**：
- 自动恢复减少人工干预需求
- 降低运维成本，提高问题解决效率

## 二、错误恢复状态机

### 2.1 状态机设计概念

错误恢复状态机定义了错误从检测到解决的完整状态流转：

**核心状态**：
- **Detected (检测)**: 初始状态，错误被检测到
- **Classify (分类)**: 对错误进行分类与评估
- **Auto (自动恢复)**: 可自动恢复的错误
- **SemiAuto (半自动恢复)**: 需要用户确认
- **Manual (人工干预)**: 需要人工介入
- **Resolved (已解决)**: 错误成功恢复
- **Exhaust (重试耗尽)**: 重试次数耗尽
- **NonRecover (不可恢复)**: 无法恢复的错误
- **Shutdown (安全停机)**: 进入安全停机流程

### 2.2 状态转换规则

| 当前状态 | 目标状态 | 触发条件 |
|----------|----------|----------|
| Detected | Classify | 错误到达恢复管理器 |
| Classify | Auto | 错误可自动恢复 |
| Classify | SemiAuto | 需要用户确认 |
| Classify | Manual | 需要人工干预 |
| Auto(Retry) | Resolved | 重试成功 |
| Auto(Retry) | Retry(自身) | 重试失败但还有重试机会 |
| Auto(Retry) | SemiAuto | 重试次数耗尽 |
| SemiAuto | Resolved | 用户选择成功 |
| SemiAuto | Manual | 用户选择升级 |
| Manual | Resolved | 人工修复成功 |
| Manual | NonRecover | 无法修复 |
| NonRecover | Shutdown | 进入安全停机 |

## 三、重试策略设计

### 3.1 重试策略核心概念

重试是处理瞬态错误的基本策略，包含以下五个设计要素：

**要素 1: 最大重试次数 (max_attempts)**
- 控制重试的总次数
- 推荐默认值: 3次

**要素 2: 初始延迟 (initial_delay_ms)**
- 首次重试前的等待时间
- 推荐默认值: 500ms

**要素 3: 最大延迟 (max_delay_ms)**
- 重试延迟的上限，防止无限等待
- 推荐默认值: 30秒

**要素 4: 退避策略 (backoff_strategy)**
- 控制延迟的增长方式
- 固定延迟: delay = initial
- 线性延迟: delay = initial + n * step
- 指数延迟: delay = initial * multiplier^n

**要素 5: 抖动因子 (jitter_factor)**
- 防止多客户端同时重试造成惊群效应
- 公式: delay = base ± (base * jitter_factor)
- 推荐默认值: 0.1 (10%)

### 3.2 指数退避算法

指数退避策略通过递增的时间间隔来避免频繁重试：

**计算公式**：
```
delay_n = min(initial_delay * multiplier^n, max_delay)
```
其中 n 为重试次数（从0开始）

**示例配置**：
- initial_delay_ms = 500
- multiplier = 2.0
- max_delay_ms = 30000

| 重试次数 | 基础延迟 | 抖动范围 (±10%) | 实际等待区间 |
|----------|----------|----------------|--------------|
| 0 | 500ms | ±50ms | [450ms, 550ms] |
| 1 | 1000ms | ±100ms | [900ms, 1100ms] |
| 2 | 2000ms | ±200ms | [1800ms, 2200ms] |
| 3 | 4000ms | ±400ms | [3600ms, 4400ms] |
| 4 | 8000ms | ±800ms | [7200ms, 8800ms] |
| 5 | 16000ms | ±1600ms | [14400ms, 17600ms] |
| 6 | 30000ms | ±3000ms | [27000ms, 33000ms] |

### 3.3 可重试错误判断

**可重试错误的特征**：

1. **瞬态错误**
   - 网络超时（网络可能很快恢复）
   - 连接被拒绝（服务可能正在启动）
   - 资源暂时不可用

2. **服务端错误 (5xx)**
   - 500 Internal Server Error (服务端临时问题)
   - 503 Service Unavailable (服务可能很快恢复)
   - 429 Rate Limited (限流，稍后重试可能成功)

**不可重试错误的特征**：

1. **客户端错误 (4xx)**
   - 400 Bad Request (请求本身有问题，重试不会解决)
   - 401 Unauthorized (认证问题，重试不会解决)
   - 403 Forbidden (权限问题，重试不会解决)

2. **业务逻辑错误**
   - 用户输入错误
   - 配置错误
   - 数据不一致

3. **资源耗尽 (永久性)**
   - 磁盘空间永久不足
   - 配额永久用尽

## 四、熔断器模式

### 4.1 熔断器概念

熔断器是一种防止级联失败的模式，通过监控服务调用失败率来自动切换状态：

**熔断器三态模型**：

**CLOSED (关闭状态)**：
- 请求正常通过
- 失败计数累加
- 当失败数达到阈值时切换到 OPEN

**OPEN (打开状态)**：
- 请求立即失败，不发往服务端
- 冷却期超时后进入 HALF-OPEN

**HALF-OPEN (探测状态)**：
- 允许探测请求通过
- 探测成功则关闭熔断器
- 探测失败则重新打开熔断器

### 4.2 熔断器参数

| 参数 | 默认值 | 含义 |
|------|--------|------|
| failure_threshold | 5 | 连续失败多少次后打开熔断器 |
| cooldown_duration_sec | 60 | OPEN状态持续多久后进入HALF-OPEN |
| half_open_max_probes | 1 | HALF-OPEN状态下允许的探测请求数 |
| success_threshold | 1 | HALF-OPEN状态下多少次成功才关闭熔断器 |

## 五、优雅降级模式

### 5.1 降级策略概念

优雅降级是在组件失败时提供有限服务的能力：

**Level 1: 立即降级**
- 激活备用资源或功能
- 对用户透明或仅有轻微延迟
- 示例: 切换到备用服务器

**Level 2: 功能降级**
- 禁用非核心功能，保留核心功能
- 用户体验部分受影响
- 示例: AI功能不可用时保留基础编辑功能

**Level 3: 缓存降级**
- 使用缓存数据提供只读服务
- 用户感知到功能受限
- 示例: 显示上次成功加载的数据

**Level 4: 安全停机**
- 保存现场，优雅退出
- 防止数据损失或损坏
- 示例: 核心数据损坏时的保护性停机

### 5.2 降级决策框架

**降级决策考虑因素**：

1. **故障组件的关键程度**
   - 核心组件 → 优先降级
   - 辅助组件 → 可接受完全失败

2. **降级方案的可用性**
   - 是否有备用方案
   - 备用方案的容量是否足够

3. **用户影响**
   - 降级后用户体验是否可接受
   - 是否需要通知用户

4. **恢复概率**
   - 故障是否是瞬态的
   - 自动恢复的可能性

## 六、回滚机制设计

### 6.1 回滚层级

**Level 1: 操作级回滚**
- 范围: 单次操作副作用
- 成本: 低 (μs~ms)
- 机制: Command Pattern Undo
- 示例: 文件操作回滚、数据库事务回滚

**Level 2: 会话级回滚**
- 范围: 当前会话所有变更
- 成本: 中 (ms~s)
- 机制: Snapshot-based Rollback
- 示例: 会话状态恢复到某个时间点

**Level 3: 系统级回滚**
- 范围: 项目/工作区级别
- 成本: 高 (取决于数据量)
- 机制: Backup Restoration
- 示例: 从备份恢复整个项目

### 6.2 回滚安全规则

**规则 S1: 幂等性保证**
- 重复回滚结果一致
- 实现 CAS (Compare-And-Swap) 语义

**规则 S2: 回滚前备份**
- 创建 pre-rollback-snapshot
- 保留 ≥ 7天

**规则 S3: 回滚失败安全网**
- 二次错误立即中止
- 进入 Safe Mode

**规则 S4: 事务语义**
- WAL 两阶段提交 (Prepare → Commit)
- 全部成功或全部回滚

## 七、最佳实践

### 7.1 设计原则

**原则 1: 尽早检测，优雅处理**
- 错误发现越早，损失越小
- 优雅处理优于粗暴崩溃

**原则 2: 区分瞬态和永久错误**
- 瞬态错误 → 重试
- 永久错误 → 快速失败，避免无效重试

**原则 3: 提供多种恢复路径**
- 不要只依赖单一恢复机制
- 自动 → 半自动 → 人工，逐级升级

**原则 4: 保护系统稳定性**
- 使用熔断器防止级联失败
- 限制重试次数和频率
- 设置超时避免无限等待

### 7.2 常见陷阱

**陷阱 1: 重试风暴**
- 问题: 大量客户端同时重试导致服务端过载
- 解决: 使用指数退避 + 抖动

**陷阱 2: 无限重试**
- 问题: 永久错误被无限重试
- 解决: 设置最大重试次数

**陷阱 3: 忽略超时设置**
- 问题: 重试时没有超时，可能永远等待
- 解决: 每次重试都设置超时

**陷阱 4: 副作用未处理**
- 问题: 操作有副作用但失败后未回滚
- 解决: 设计回滚机制，记录操作历史

## 八、总结

错误恢复设计是构建可靠系统的重要组成部分。一个好的错误恢复系统应该：

1. **多层次恢复**：支持自动、半自动、人工等多种恢复方式
2. **智能重试**：正确识别瞬态错误，使用指数退避避免惊群
3. **熔断保护**：防止级联失败，保护系统整体稳定性
4. **优雅降级**：在部分失败时提供有限服务
5. **安全回滚**：支持操作回滚，保护数据一致性
6. **可观测**：恢复过程全程记录，便于分析

本文档提供的错误恢复设计模式是通用指导，具体实现应根据业务场景、可靠性要求和资源成本进行调整。
